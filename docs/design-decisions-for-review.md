# Design Decisions For Review

This document is the compact review ledger for `stream_v3`. It does not replace
the detailed runtime docs, but it should give a reviewer enough design context to
understand the system as a reliability and platform engineering case study.
It includes current v3 decisions and the v1 `/stream` / `stream_v2` decisions
that still explain why v3 is shaped this way.

The most important pattern is repeated throughout the project: delivery,
observation, source data, recovery, and public presentation are separate
failure domains. A dashboard warning is not automatically a delivery failure,
and a delivery symptom is not automatically permission to mutate YouTube state.
For reviewers who want to verify the claims against implementation and tests,
`docs/implementation-review-map.md` is the shortest path.

Origin notes for reviewers:

- `/stream` and `stream_v2` rows are inherited constraints from real incidents,
  not nostalgia for an older stack.
- `stream_v3` rows are the k3s/split-plane implementation of those constraints.
- Migration rows explain that a workload being healthy is not the same as
  transferring production authority.
- Rows that say v2 remained production refer to the migration window. In the
  current public snapshot, v3 is the production delivery shape after explicit
  cutover evidence.

## Decision Ledger

| Area | Decision | Why it was chosen | Alternative rejected | Trade-off or review note |
| --- | --- | --- | --- | --- |
| Runtime boundary | Use k3s for `stream_v3` delivery. | The media workload needed a clearer boundary after browser rendering, audio, FFmpeg, watchdogs, and recovery competed inside earlier single-host ownership. | Continue with systemd-only ownership for the whole stack. | Kubernetes adds operational complexity, but makes container ownership, resource limits, state mounts, and dry-run validation explicit. |
| Runtime ownership | Keep `stream-v3-runtime` as a 3-container delivery pod: `stream-engine`, `auto-dj`, and `fast-recovery-loop`. | Rendering, music, and local recovery have different failure modes but share delivery-plane state. | Run all delivery logic as one process or move DJ/recovery back to host services. | More container wiring, but container restart counts and logs identify which delivery component moved. |
| Single publisher guard | Prevent multiple live publishers for one ingest target. | Legacy `/stream` multi-start produced competing publishers for the YouTube primary ingest URL. | Let stream processes race and rely on YouTube to choose one. | Runtime locks, replica limits, and restart attribution add ceremony, but recovery must not create a second publisher. |
| Ingest secret contract | Validate stream-key and output configuration before publishing, and mask secrets in logs. | Legacy placeholder or manual env mistakes could fail startup or expose the ingest URL. | Trust manual exports and log the full RTMP/RTMPS destination. | More startup checks, but public evidence and incident logs stay safe to share. |
| Delivery / observability | Split delivery and observability. | Monitoring should classify evidence and request recovery without directly owning FFmpeg. | Run delivery, monitoring, and recovery on the same host with shared process ownership. | More coordination and remote evidence plumbing, but smaller recovery blast radius. |
| Observability host | Keep HP ProDesk as the observability owner. | Long-window SLI, dashboard state, YouTube watchdog state, notifications, and staged recovery decisions should survive delivery-host pressure. | Let the Dell delivery host own all monitoring loops. | Requires remote runtime evidence, but prevents local delivery pressure from blinding monitoring. |
| Public gateway | Keep Raspberry Pi as a gateway, not the metrics backend. | The public status UI and `/grafana/` proxy can be exposed without moving Prometheus/Loki/Grafana state. | Move the monitoring backend to the Raspberry Pi. | Simpler public entrypoint, while evidence retention stays on the observability host. |
| ADS-B source boundary | Keep the Airspy/readsb source chain distinct from `stream_v3` delivery. | Airspy on HP ProDesk, `airspy_adsb`, ProDesk readsb, Dell readsb, and the Dell modified tar1090 map endpoint can fail differently from browser rendering or RTMPS delivery. | Treat the ADS-B map endpoint as just another renderer symptom. | Source freshness checks become explicit, but public k3s manifests do not attempt to own the Airspy device. |
| Rendering source | Render from readsb/tar1090/custom map data, not from another stream output. | The stream should produce its own video from source data rather than re-encode or depend on an existing stream. | Use another stream output as a secondary input. | More browser/rendering ownership, but clearer source-to-video evidence. |
| Browser capture contract | Keep browser/Xvfb capture as the rendering boundary for JS-driven map output. | The modified tar1090 map is a browser-rendered product, so the stream must capture the viewer UI that actually exists. | Feed map pages directly into FFmpeg or replace the UI with a non-browser source. | Browser capture is heavier, but it preserves the real map, overlay, labels, and visual defects. |
| Startup readiness | Gate FFmpeg start on local rendering/audio readiness instead of folklore fixed sleeps. | `/stream` measurements showed overlay readiness, not a 20-25 second fixed wait, was the local startup constraint. | Sleep a large fixed amount before every FFmpeg start. | Probe-driven readiness shortens recovery, but the probes must be kept meaningful. |
| Viewer UI boundary | Keep internal debug/status text out of the viewer overlay. | Legacy incidents exposed stale/debug fields and browser translation UI in the public frame. | Let operational state leak into the stream for convenience. | Operators need dashboards and capture evidence; the stream itself stays viewer-facing. |
| Actual range guard | Validate overlay supplement data before merging it into the map range. | One bogus aircraft point can distort the displayed actual range for many hours. | Trust every persisted supplement point. | Conservative pruning can reject rare true outliers, but avoids visible range spikes. |
| Fast recovery | Keep fast recovery in the delivery plane at a short interval. | RTMPS TCP samples and local FFmpeg restart decisions are closest to the delivery process. | Let the observability monitor directly own FFmpeg restart loops. | Faster local reaction, while observability host still owns classification and notification. |
| Recovery cadence | Prefer short local recovery cadence over textbook exponential backoff. | Same URL recovery opportunities are time-sensitive; waiting longer after each failed probe can turn a recoverable RTMPS gap into viewer-facing drift. | Use exponential backoff primarily to reduce network/probe load. | Load is controlled by evidence gates, restart budgets, and quota guards instead of slowing local reaction. |
| Recovery safety | Stage recovery through guards. | False restarts or YouTube mutations can damage a live stream more than a short observation delay. | Let watchdogs restart or mutate immediately on local symptoms. | Recovery may be slower, but destructive actions require fresher and more consistent evidence. |
| Restart budget | Bound recovery-induced downtime with hourly and daily restart budgets. | Restarts are useful, but repeated noisy restarts can become the incident. | Restart on every warning, or block restarts whenever the network looks bad. | Budgeting preserves fast recovery while making recovery cost visible. |
| FFmpeg child recovery | Let the stream engine restart only the FFmpeg child for recoverable transport failures. | Broken pipe / exit 224 can recover faster than restarting browser, PulseAudio, AutoDJ, and the whole runtime. | Treat every FFmpeg exit as a whole-stream failure. | Child recovery needs its own SLI so it is not hidden by current health. |
| Bounded FFmpeg stop | Stop stuck FFmpeg inside the engine before the supervisor force-kills the parent. | TCP stalls can make graceful FFmpeg shutdown exceed service or Pod stop timeouts. | Increase stop timeout and wait longer. | Faster recovery and better JSONL evidence, at the cost of less graceful muxer close. |
| Recovery profiles | Apply temporary low-upload profiles only after strong transport recovery evidence. | During route or modem instability, short-lived bitrate reduction can protect live continuity. | Drive profile changes directly from raw Mbps, public-probe noise, or dashboard warnings. | Visual quality can dip briefly, but control input stays tied to a strong network-down event. |
| Restart attribution | Record restart requests before manual rollout, staged restart, or observability-host remote recovery. | A Kubernetes `restartedAt` annotation proves that a Pod template changed, but not who requested it or why. | Reconstruct restart cause later from audit logs, ReplicaSets, and operator memory. | Adds a small wrapper/state-file requirement, but future Pod replacement can be classified as manual, automatic, or fault. |
| Production authority | Transfer production authority only through explicit cutover evidence. | During migration, v2 remained production until systemd, CLI, state root, monitoring, and mutation paths were intentionally moved. | Treat a Running v3 Pod or green shadow check as ownership transfer. | Cutover takes more proof, but v2 safety gates are not bypassed by infrastructure readiness. |
| Migration smoke test | Use a 24-hour live smoke test for v3 runtime, encoder, recovery, or cutover-authority changes. | v2 already established the stable behavior model; v3 still has to prove migrated ownership across one daily cycle. | Treat a short launch test or green Pod as enough. | A 24-hour pass is migration confidence, not a long-window SLO claim. |
| Shadow mode | Keep shadow mode before cutover and retain it as a non-mutating validation path. | The system should prove command mapping, state writes, action plans, and guard behavior before production mutation. | Enable k3s recovery actions directly before cutover evidence exists. | More validation steps, but clearer safety evidence. |
| Cutover flags | Require explicit production flags for live mutation. | `STREAM_V3_MODE`, cutover enablement, dry-run state, and test mode should make accidental mutation hard. | Infer production behavior from environment shape alone. | More configuration surface, but a safer operator contract. |
| Maintenance mode | Treat planned work as maintenance state, not ordinary incident spam. | Some restarts, checks, and temporary warnings are operator-intended work. | Let monitors page as if every planned change is an outage. | Requires explicit maintenance state, but notifications and SLO interpretation are clearer. |
| Same URL | Treat same-URL preservation as a first-class SLO. | Losing the public YouTube URL is more damaging than many short local faults. | Replace broadcasts aggressively whenever YouTube evidence is ambiguous. | Recovery must preserve identity unless replacement is explicitly justified. |
| Persistent broadcast | Prefer preserving a validated persistent scheduled broadcast over elapsed-time replacement. | `enableAutoStop=false` changed the tradeoff: elapsed time alone does not prove the current URL is unrecoverable. | Replace after a fixed 180 second window. | Replacement remains a fallback, but only after identity and recoverability checks. |
| YouTube identity | Resolve and monitor the current video ID through resolver/watchdog state. | Same URL, public live state, OAuth/Data API state, and candidate replacement evidence must be correlated. | Trust a single source such as the watch page, Data API, or cached URL. | More state and cache handling, but fewer false replacements. |
| YouTube mutation gate | Require OAuth channel match, evidence ledger state, and an action gate before destructive YouTube actions. | Public page, Data API, OAuth, and cached state can disagree during outages. | Let watchdog code directly transition, bind, or replace broadcasts. | More machinery, but wrong-channel or stale-evidence mutations are blocked. |
| OAuth control plane | Treat OAuth token health as control-plane evidence, not delivery health by itself. | `invalid_grant` or token rotation can degrade mutation authority while RTMPS ingest and public live remain healthy. | Restart delivery or replace broadcasts because an OAuth probe fails. | Operators must repair credentials without confusing the control plane with viewer impact. |
| Resolver fast mode | Use fast resolver cadence only under unhealthy or confirmed conditions, and keep configured IDs as boot seeds only. | v2 quota incidents showed high-cost search and stale configured IDs can amplify outages. | Search frequently or fall back to configured video IDs during normal operations. | Recovery stays fast when needed without burning quota or reviving stale IDs. |
| Quota-aware aggressiveness | Spend a measured quota budget on faster unhealthy-state resolution. | Routine checks showed about 1.5k-1.8k YouTube API units/day against a 10k/day quota envelope and a lower alert threshold. | Slow all YouTube checks to conserve quota, or call expensive API search every 5 seconds. | Fast mode stays aggressive where it matters, while high-cost calls are confirmed and gated. |
| Public live proof | Separate public live availability from local ingest. | FFmpeg/RTMPS connected does not prove that viewers see a live watch page. | Treat ingest connected as proof of public live. | Public probes can be noisy, so negative public evidence alerts before destructive action. |
| API quota | Treat API quota exhaustion as degraded evidence, not delivery failure. | YouTube API failure should not automatically imply stream failure if public/local evidence remains good. | Treat API errors as authoritative stream failure. | State classification is more complex, but quota pressure is less likely to cause bad recovery. |
| API day accounting | Display PT open-day actual units separately from projected/day and closed-day history. | YouTube quota resets on Pacific Time and dashboard top-line numbers must not confuse actual usage with projection. | Show projected/day or latest closed-day units as current-day usage. | More metrics, but the operator can tell quota risk from actual burn. |
| Quota guard partition | Prefer transport, public, and local evidence when quota exhaustion degrades control-plane quality. | 403 responses can make Data API/OAuth state less authoritative. | Continue treating quota-exhausted API reads as normal lifecycle truth. | Decisions become source-weighted instead of API-first. |
| Health model | Judge current health from multiple signals, not one dashboard stat. | Delivery status needs YouTube live, ingest, Pod state, capture, audio, upload, watchdog, notify, and freshness evidence. | Let `Current`, `Network`, `Watchdog`, or any single panel decide health. | Slightly slower diagnosis, but fewer restarts from stale or mis-scoped panels. |
| Dashboard interpretation | Classify dashboard anomalies as current incident, historical event, or observability noise. | Long-window or stale series can display `FAIL` or `CHANGED` after the live system has recovered. | Treat every visible dashboard failure as current delivery impact. | Operators must inspect raw metrics, but error budget is not burned by false positives. |
| Active notification gate | Notify current incidents only from fresh current-fail signals, not historical degraded windows. | v2 Discord loops made recovered RTMPS/public-probe history look active. | Page on every 1h/24h warning count. | Historical degradation remains in routine checks, while alerts describe current impact. |
| Report-only promotion | Keep weak observability probes report-only until repeated low-false-positive correlation exists. | Map movement samples, public probes, TLS strings, needrestart checks, and upstream reports are useful but noisy. | Wire every new probe directly to restart or alert control. | Some real weak signals wait for review, but recovery stays stable. |
| Metrics namespace | Use `stream_v3_*` metrics and `job="stream_v3_observability_monitor"` as the v3 source of truth. | v2-compatible names made the v3 dashboard ambiguous. | Keep v2 metric names and rely on labels or dashboard titles. | Requires exporter/dashboard migration, but source ownership is visible in queries. |
| Dashboard query isolation | Keep production and shadow metric queries scoped by job/source. | A shadow v3 source can otherwise make a v2 production dashboard red through a broad `max()` query. | Aggregate v2 and v3 compatible names without source filters. | Dashboards need more explicit queries, but red panels point to the right owner. |
| Exporter shape validation | Test exporter mappings against the actual health-summary JSON shape. | A panel can be wrong if the exporter reads `window.pass` when the real value lives under `window.observe.pass`. | Trust dashboard panels without checking raw JSON and raw series. | Contract tests and raw metric checks are extra work, but prevent believable false health states. |
| Remote TCP evidence | Use fresh remote TCP send samples for ingest/network health. | The observability host cannot directly inspect in-Pod FFmpeg sockets, so local socket probes can lie. | Mark network failed whenever the observability-side observer cannot see the Pod socket. | More exporter logic, but dashboard network state reflects delivery reality. |
| Network observer boundary | Keep route, DNS, IPv4/IPv6, and TCP-connect cause layers observer-only until correlation is proven. | Cause-layer probes explain recurrence but can be noisy and endpoint-dependent. | Promote local-route, DNS, or remote-connectivity labels directly to recovery triggers. | Evidence improves without expanding recovery blast radius. |
| Upload SLI | Use windowed TCP samples for upload latest, p95, max, age, and over-budget seconds. | Old summary fields could misrepresent `Upload p95 1h`. | Reuse older 24h health fields for dashboard upload panels. | More metrics, but tuning decisions match observed RTMPS send behavior. |
| SLI window semantics | Separate rolling, cumulative, and regime-bounded SLI. | Old unstable periods made cumulative ratios look like current quality. | Show one lifetime percentage as the health number. | More labels, but current regime and historical record stop fighting each other. |
| Restart SLI aggregation | Report FFmpeg restarts as attempts, retry episodes, incident clusters, root cause, and max duration. | Raw restart counts overstate independent incidents. | Count each FFmpeg restart as one incident. | More complex aggregation, but operators see retry density and impact. |
| SLO policy | Track availability, same URL, YouTube input quality, upload ceiling, visual correctness, and audio correctness separately. | Upload efficiency alone is not the product outcome. | Treat low upload as always good or treat all warnings as equal. | The operator must say what changed and what did not change when budget burns. |
| Subsystem SLI boundary | Keep ADS-B freshness, API quota, notify state, visual, audio, upload, and availability as distinct signals. | A live YouTube stream can still have stale ADS-B data, muted audio, noisy alerts, or quota-degraded evidence. | Collapse every subsystem into one availability percentage. | Reviewers see more numbers, but each number has a clear owner and action path. |
| SLO priority | Prioritize availability and same URL over input quality, then visual/audio correctness, then upload efficiency. | Lower upload settings are not acceptable if they create YouTube input warnings. | Optimize first for bandwidth. | Upload may sit in the high 4 Mbps range because YouTube health matters more. |
| RTMPS transport | Use RTMPS/443 as the accepted YouTube ingest transport. | The live path should use YouTube's encrypted ingest endpoint and be checked by contract tests. | Stay on legacy RTMP as the normal path. | RTMPS adds TLS-specific failure modes and upload accounting differences, so rollback and monitoring stay explicit. |
| TLS observability | Track RTMPS SSL/TLS-specific events separately from generic Broken pipe or tcp_stall. | SSL/TLS failures need different triage than ordinary transport resets. | Collapse all transport errors into exit 224 or network fail. | This does not restart by itself; it gives sharper follow-up evidence. |
| A/V timestamp contract | Keep FFmpeg timestamp stabilization as part of the encoder contract. | Legacy x11grab and PulseAudio capture produced non-monotonic DTS/PTS warnings. | Treat timestamp warnings as harmless long-running noise. | Extra FFmpeg arguments make the command less minimal, but long-running muxing is steadier. |
| Encoder baseline | Use NVIDIA NVENC as the production encoder. | The v3 production host has GPU capability and the runtime contract requires GPU encode. | Return to CPU `libx264` as the normal production path. | CPU encode remains a debug/fallback path, not the target. |
| Encoder rate control | Use `h264_nvenc` CBR for production. | VBR/CQ reduced upload but caused YouTube low bitrate / not enough video warnings. | Keep VBR/CQ because it used less bandwidth. | More bandwidth, but better YouTube input quality. |
| Encoder cadence | Keep 5fps/3400k/audio192k as the current env-synced v3 encoder contract. | A 2026-05-31 4fps/5fps/10fps comparison held 3400k video CBR constant; 5fps stayed under the upload ceiling while improving cadence with a smaller per-frame quality loss than 10fps. | Stay on 4fps after YouTube low-video warnings, or promote 10fps because it was smoother in a short trial. | YouTube input quality remains monitored separately, and contract drift checks must use 5fps/3400k. |
| Encoder lineage | Keep v2, v3 trials, and current v3 runtime named separately. | "Current FPS" changed meaning across the migration and tuning work: v2 production ended in the low-fps regime, v3 tested 30fps, then v3 compared 4fps/5fps/10fps and accepted 5fps/3400k. | Describe the encoder as one continuous contract. | Reviewers can see 5fps to 4fps in v2, the 30fps experiment, and the measured 5fps v3 contract without treating those as contradictions. |
| Encoder efficiency | Do not promote NVENC quality knobs to current production without env sync evidence. | `p5`, fullres multipass, lookahead, AQ, and B-frames were useful trial knobs, but current env authority is the p4 CBR contract. | Leave trial settings in the live ConfigMap because they once improved input quality. | Simpler runtime contract, while future quality changes still require YouTube health and upload evidence. |
| Upload ceiling | Treat 5.0 Mbps as the warning ceiling, not as the tuning target. | Normal operation should stay below the line while retaining YouTube health. | Force every profile into 4.0-4.5 Mbps regardless of warnings. | Some samples can approach 4.8-4.9 Mbps, but input quality remains protected. |
| Audio transport | Start container PulseAudio with `--disable-shm=yes` and `--enable-memfd=no`. | PulseAudio shm/memfd behavior produced container audio failures. | Use default PulseAudio shared-memory transport. | Sacrifices an optimization to make the audio route stable. |
| Audio quality budget | Keep music audio quality as part of the product, within upload budget. | v2 source-audio review showed 192k audio had small upload cost relative to the video stream and matched most source files better. | Minimize audio bitrate because it is not the ADS-B map. | Slightly higher upload, but music remains a deliberate viewer-facing feature. |
| Audio SLI | Verify audio by Pulse route, sink/source presence, sink inputs, monitor energy, and audio fault counters. | A running DJ process does not prove audio is reaching FFmpeg. | Treat process liveness as audio health. | More probes, but silent-stream failures are easier to catch. |
| Audio recovery restraint | Stage audio recovery and suppress track-transition noise. | Song changes and Pulse probes can look like silence and can amplify faults. | Restart DJ or stream on the first low-energy sample, or run intrusive probes constantly. | Silent-stream detection remains, but recovery waits for repeated evidence. |
| Visual SLI | Verify visual correctness with capture evidence. | Browser/rendering can fail while YouTube ingest remains connected. | Treat FFmpeg or YouTube ingest as proof of visual correctness. | Requires capture checks, but map, aircraft, receiver panel, now-playing, footer, and error band are visible evidence. |
| Visual report-only boundary | Treat map/upstream visual probes as report-only unless viewer impact is repeated and correlated. | Low-traffic 5 second aircraft-movement samples can look broken for a whole report interval. | Restart the stream or page active incidents on every movement-only warning. | Some upstream visual issues require human review, but observation noise does not mutate delivery. |
| Now playing | Treat now-playing metadata as runtime evidence. | `Unknown title` is a real viewer-facing defect and also indicates bridge/state problems. | Ignore music metadata as cosmetic. | Adds small state checks, but catches overlay regressions. |
| Music rotation evidence | Keep AutoDJ play history and rotation classification as evidence, not just playlist behavior. | Legacy AutoDJ issues included skipped playback, major/minor exhaustion, and missing audit history. | Treat music as an opaque background process. | More state files, but now-playing and rotation regressions can be investigated. |
| Runtime memory guard | Make `stream_v3_runtime_memory_*` measure the `stream-v3-runtime` Pod, not the observability host. | Host memory pressure and delivery Pod memory are different failure domains. | Use HP ProDesk host memory as the delivery memory guard. | Requires Kubernetes metrics API access, but avoids misleading Grafana signals. |
| Xvfb memory guard | Add a process-level guard for Xvfb RSS/shared memory. | A prior OOM was caused by Xvfb shared-memory growth that container current-memory samples could miss between scrapes. | Rely only on Prometheus container memory samples. | More in-container inspection, but prevents a full stream-engine OOM path. |
| Memory attribution | Classify anonymous memory, file cache, inactive file, one-shot peaks, and non-reclaimable estimate separately. | AutoDJ file cache and one-shot systemd peaks can look large without being leaks or current pressure. | Use `htop` used memory or service current as the budget. | Memory reports are more verbose, but fewer false memory incidents are created. |
| Memory action policy | Do not restart only because memory looks large. | Shared memory and RSS can look high without cgroup OOM/high/max events or guard breach. | Restart whenever Grafana memory looks visually large. | Operators must inspect trend, limit ratio, events, and process breakdown. |
| Host freeze recovery | Add host watchdog policy for the single-node delivery host. | In-Pod recovery cannot recover a frozen OS, PID1 stall, or kernel lockup. | Assume k3s or fast recovery can handle all delivery failures. | `softdog` and systemd watchdog improve recovery, but hardware watchdog still depends on BIOS support. |
| DR model | Document single-node DR and rollback separately from live recovery. | Node rebuild, host reboot, v2 rollback, and live FFmpeg restart have different risk and RTO. | Use one generic recovery playbook. | More runbooks, but failure response is scoped. |
| Legacy rollback | Keep older repo/state/env rollback paths explicit, but not automatic. | v1/v2 cutovers changed unit files, CLI symlinks, state roots, and env ownership. | Delete older runtime roots immediately or auto-fallback on faults. | More archived surface, but rollback does not depend on operator memory. |
| Third-party validation | Treat runbook third-party dry-run as a quality gate. | A runbook that only the author can execute is not operationally mature. | Accept docs after the original operator verifies them. | Adds review work, but surfaces hidden assumptions. |
| Failure taxonomy | Name failures by owner, detection, action, and evidence. | `FAIL` is too vague across delivery, source, YouTube, dashboard, and monitor layers. | Collapse symptoms into generic stream failure. | More vocabulary, but triage is faster and safer. |
| Runtime JSONL retention | Rotate raw runtime JSONL, read current plus rotated/gz files, and promote long-term evidence into docs. | 1h/8h/24h SLI needs replay, but raw logs cannot grow forever. | Keep raw JSONL indefinitely, or only read latest snapshots. | Long-term analysis relies on summarized docs after retention expires. |
| Documentation responsibility | Separate current contracts, runbooks, decisions, incidents, tuning, routine checks, ops logs, next plans, and legacy. | v2 ops logs had started carrying evidence, current guidance, procedures, and future plans at once. | Leave all judgment inside chronological logs. | More document maintenance, but reviewers can distinguish history from current rules. |
| Evidence retention | Keep runtime state and secrets out of Git, but keep evidence references in ops docs. | Reviewers need reasoning without publishing stream keys, tokens, webhook URLs, or runtime state. | Commit state files or omit evidence entirely. | Public docs are less directly replayable, but the security boundary is preserved. |
| Public CI | Keep public CI small and non-mutating. | The public repository should prove snapshot integrity without secrets, real RTMPS, or cluster mutation. | Run full live pytest, YouTube checks, and k3s apply in CI. | CI proves fewer production behaviors, but avoids unsafe environment-specific checks. |
| Public release scope | Present the repository as a case study, not a reusable streaming product. | The value is the reliability design, operational evidence, and safety model, not a generic installer. | Package it as a plug-and-play YouTube streaming stack. | Reviewers see real engineering judgment without assuming their environment can run it unchanged. |

## Review Focus

When reviewing the project, the highest-signal questions are:

1. Are delivery, observation, ADS-B source, public gateway, and recovery
   ownership boundaries clear?
2. Do destructive actions require fresh, consistent, identity-preserving
   evidence?
3. Are dashboard failures cross-checked against raw runtime metrics before they
   become incidents?
4. Do SLO decisions explain what changed and what intentionally did not change?
5. Are inherited v1/v2 incidents represented as durable design constraints
   rather than copied as raw incident history?
6. Are the public docs and tests honest about what can be validated without
   credentials, production state, or live mutation?
